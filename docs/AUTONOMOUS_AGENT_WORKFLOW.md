# Автономный агент: общие инструкции

Этот документ задаёт общий режим работы автономного агента без жёсткой привязки к конкретной реализации API.

## Цель

Агент должен работать циклично и полностью автономно:

1. запускать процесс автооткликов;
2. ждать завершения;
3. обрабатывать последние 15 диалогов;
4. генерировать и отправлять ответы;
5. повторять цикл по расписанию.

## Базовый цикл

1. Проверить, что профиль готов к работе (авторизация, токен, доступ к данным).
2. Запустить задачу автооткликов.
3. Дождаться завершения задачи и сохранить результат в лог.
4. Получить последние 15 актуальных диалогов.
5. Для каждого диалога:
   - получить свежий контекст;
   - сформировать ответ;
   - применить правила качества;
   - отправить сообщение;
   - обновить checkpoint состояния.
6. Завершить цикл и перейти в ожидание следующего запуска.

## Принципы автономности

- Не требовать ручного вмешательства в штатном режиме.
- Уметь восстанавливаться после временных ошибок (retry с лимитом попыток).
- Не запускать второй цикл, пока не завершился первый (lock/lease).
- Поддерживать идемпотентность: повторный запуск не должен дублировать действия.

## Правила ответов в диалогах

- Тон: вежливый, деловой, краткий.
- Язык: русский (если контекст не требует иного).
- Не обещать того, что не подтверждено пользователем.
- Не отправлять пустые или слишком короткие ответы.
- При неуверенности использовать безопасный нейтральный шаблон.

## Антидубли и состояние

Агент должен хранить служебное состояние по каждому диалогу:

- последний обработанный входящий message_id (или timestamp);
- время последней отправки;
- статус последней попытки.

Правило: если новых входящих сообщений нет, диалог пропускается.

## Ошибки и эскалации

- Временные ошибки сети/сервиса: retry с backoff.
- Ошибки авторизации: отдельный статус и алерт.
- Критические ошибки (невозможен запуск цикла): остановка текущего прогона + уведомление.
- Частичные ошибки по отдельным диалогам не должны останавливать весь цикл.

## Наблюдаемость

Для каждого цикла логировать:

- старт/финиш и длительность;
- сколько откликов инициировано;
- сколько диалогов обработано из 15;
- сколько ответов отправлено;
- сколько пропущено и почему;
- какие ошибки произошли.

## Режим запуска

- Запуск по расписанию (например, каждые 15–30 минут).
- Добавлять небольшой случайный сдвиг старта (jitter), чтобы избежать “ровного” паттерна.
- Отдельно хранить конфиг агента и состояние (state), чтобы можно было безопасно перезапускать процесс.

## Границы ответственности

Агент сам определяет технические детали вызовов и интеграции, но должен сохранять этот контракт поведения:

- сначала отклики, потом ответы по сообщениям;
- в каждом цикле обрабатываются только последние 15 диалогов;
- действия не дублируются;
- при проблемах с авторизацией создаётся явный сигнал для ручного вмешательства.
